1. Resumen Ejecutivo

Este documento describe el proyecto Data Center Automatizado, desarrollado para el curso de Arquitectura de Computadoras I. Se trata de un sistema distribuido que usa tres Arduino Uno como esclavos y un Arduino Mega 2560 como maestro, todos conectados por el bus I2C. Un modulo ESP32 funciona como gateway WiFi y se comunica con el Mega por UART usando un convertidor de nivel TXS0108E.

El sistema monitorea tres areas: la sala de servidores (temperatura, humedad, humo), el area de acceso con puertas (sensores PIR y botones) y el jardin (humedad del suelo, temperatura ambiente y riego). El Mega se encarga de juntar toda la informacion, decidir cuando activar alarmas, controlar el ventilador por PWM y mostrar datos en una pantalla LCD 16x2. Los datos se envian al ESP32, que los transmite por WiFi a una API REST hecha en Bun con Elysia, y se guardan en una base de datos PostgreSQL en AWS.

El sistema incluye deteccion de humo con linea base adaptativa, ventilador con cuatro velocidades segun la temperatura, alarma de intrusion que solo se activa si el PIR detecta movimiento con las puertas cerradas, y un bot de Telegram para recibir notificaciones.

2. Introduccion

Los Data Centers son infraestructura critica para cualquier empresa que dependa de servicios digitales. Mantener la temperatura adecuada, controlar quien entra y sale, y detectar problemas a tiempo es fundamental para evitar perdidas de informacion o danos en los equipos. Este proyecto busca resolver esos problemas usando microcontroladores Arduino, que son accesibles y permiten aplicar los conceptos que vemos en el curso.

El objetivo fue construir un sistema que pudiera monitorear y controlar un Data Center en miniatura, con sensores reales tomando mediciones, actuadores respondiendo a las condiciones del ambiente, y todo conectado a la nube para guardar historicos. Aplicamos conceptos de arquitectura de computadoras como buses de datos (I2C, UART), jerarquia de memoria, modulacion PWM y arquitectura RISC del AVR.

El sistema se compone de un Arduino Mega como maestro I2C, tres Arduino Uno como esclavos, un ESP32 para WiFi, una API REST y PostgreSQL en AWS. Cada esclavo maneja una zona distinta del Data Center y reporta al maestro, que es el que toma las decisiones finales.

3. Marco Teorico

3.1 Arquitectura RISC del AVR

Los microcontroladores ATmega2560 (del Mega) y ATmega328P (del Uno) usan una arquitectura RISC de 8 bits. Esto significa que tienen un conjunto de instrucciones reducido, donde la mayoria se ejecutan en un solo ciclo de reloj. Cada instruccion mide 16 bits. A diferencia de procesadores CISC como los de Intel, el AVR puede ejecutar una instruccion por ciclo de reloj gracias a su pipeline de dos etapas: mientras una instruccion se ejecuta, la siguiente ya se esta cargando.

La frecuencia de reloj es de 16 MHz, lo que da un tiempo de 62.5 nanosegundos por instruccion en el mejor caso. El chip tiene 32 registros de proposito general conectados directamente a la ALU, lo que acelera las operaciones aritmeticas. Esta arquitectura es ideal para sistemas embebidos en tiempo real como este proyecto.

3.2 Jerarquia de Memoria

En el sistema manejamos varios niveles de memoria, igual que en una computadora normal:

  • Flash: 256 KB en el Mega, 32 KB en cada Uno. Ahi se guarda el programa y nunca se pierde al apagar.
  • SRAM: Solo 8 KB en el Mega y 2 KB en los Unos. Es poca, asi que hay que cuidar las variables y usar PROGMEM para guardar constantes en la Flash.
  • EEPROM: 4 KB en el Mega. Se usa para guardar configuraciones como los umbrales de temperatura.
  • Base de datos en la nube: PostgreSQL en AWS. Es el nivel mas alto, con capacidad practicamente ilimitada para guardar historicos de lecturas y eventos.

3.3 Buses y Protocolos de Comunicacion

Usamos cuatro protocolos de comunicacion distintos:

I2C: Es el bus principal entre el Mega y los Unos. Usa dos cables (SDA y SCL) y permite conectar hasta 127 dispositivos. Opera a 100 kHz y cada esclavo tiene una direccion unica. El Mega es el unico maestro y los Unos son esclavos. El LCD tambien va en este bus.

UART: Se usa entre el Mega (Serial1, pines 18 y 19) y el ESP32 (Serial2, pines 16 y 17) a 9600 baudios. Necesita el TXS0108E porque el Mega trabaja a 5V y el ESP32 a 3.3V.

SPI: Lo usan los microcontroladores internamente para comunicarse con perifericos rapidos, aunque en este proyecto no lo explotamos directamente.

WiFi: El ESP32 se conecta a la red local (SSID Omega) y hace peticiones HTTP a la API.

3.4 Modulacion PWM

El control del ventilador se hace con PWM (modulacion por ancho de pulso). Consiste en prender y apagar la senal muy rapido (a 490 Hz) y variar el tiempo que esta prendida vs apagada. Esto permite controlar la velocidad del motor sin perder torque. Usamos cuatro niveles: 0 (apagado), 20%, 40% y 60% del ciclo de trabajo, que se traducen en valores de 0, 51, 102 y 153 para la funcion analogWrite().

4. Diseno del Sistema

4.1 Arquitectura General

El sistema es maestro-esclavo con bus I2C. El Mega 2560 es el unico maestro y los tres Unos son esclavos con direcciones 8, 9 y 10. El Mega hace un ciclo continuo donde le pide datos a cada esclavo, procesa la informacion, actualiza la pantalla y el buzzer, y envia todo al ESP32.

En sentido contrario, la API puede generar comandos (como abrir puertas), el ESP32 los recoge haciendo polling, los manda al Mega, y el Mega se los reenvia al esclavo correspondiente por I2C.

4.3 Flujo de Datos

Los sensores conectados a cada Uno hacen lecturas cada cierto tiempo. Cuando el Mega pide datos por I2C, cada esclavo responde con una cadena de texto. Por ejemplo, el esclavo 1 devuelve algo como "|1,27,65,0" que significa humo=1, temperatura=27 grados, humedad=65%, ventilador apagado.

El Mega recibe las tres cadenas, las procesa y arma una linea con toda la informacion separada por pipes (|). Esa linea la manda al ESP32 por el puerto serie. El ESP32 la recibe, la separa en las tres areas, y si detecta que algun valor cambio desde la ultima vez, hace un POST a la API con los datos nuevos.

Esto de solo enviar cuando hay cambios es importante para no saturar la red ni la base de datos. Por ejemplo, si la temperatura se mantiene en 27 grados durante una hora, solo se envia un POST al inicio y despues nada hasta que cambie.

5. Implementacion

5.1 Esclavo 1 - Sala de Servidores

Este esclavo tiene un DHT11 para temperatura y humedad, un MQ-135 para detectar humo, y un ventilador en el pin 6 controlado por PWM. Los datos los envia por I2C con este formato: primero el valor de humo (0 o 1), luego la temperatura, luego la humedad, y por ultimo el nivel del ventilador (0 a 3).

La deteccion de humo funciona con un algoritmo de linea base. Cuando arranca el sistema, los primeros 30 segundos se usan para calcular el valor normal del aire. Despues, si la lectura sube mas de un 5% por encima de esa linea base, se activa la alerta de humo. La linea base se va ajustando con el tiempo para adaptarse a cambios lentos del ambiente.

El ventilador tiene cuatro velocidades que se eligen solas segun la temperatura:

TemperaturaLo que hace el ventilador
Menos de 20 gradosApagado completamente
Entre 20 y 24 gradosPrende suave (20% de potencia)
Entre 25 y 29 gradosVelocidad media (40% de potencia)
30 grados o masMaxima velocidad (60% de potencia)

El motor es de 1.5V pero lo alimentamos desde el pin digital del Arduino a 5V con PWM. Para no quemarlo, nunca pasamos del 60% del ciclo de trabajo, que da un voltaje promedio de unos 3V.

5.2 Esclavo 2 - Puertas y Acceso

Este esclavo controla dos puertas con servomotores SG90. Cada puerta tiene un boton que al presionarlo la abre por 5 segundos y luego se cierra sola. Tambien tiene un sensor PIR que detecta movimiento.

Los datos que envia por I2C incluyen el estado de los botones (0 o 1), si el PIR detecto movimiento, la temperatura del DHT11 que tiene, y si cada puerta esta abierta o cerrada. Es decir: btn1, btn2, pir, temperatura, puerta1, puerta2.

El PIR tiene un sistema para evitar falsos detectores: espera 200 milisegundos para confirmar que si hay movimiento, y una vez que detecta, espera 6 segundos antes de poder detectar otra vez. Las puertas se abren solo 5 segundos y vuelven a cerrar solas.

5.3 Esclavo 3 - Jardin

Este esclavo monitorea el jardin. Tiene un sensor FC-28 que mide que tan humeda esta la tierra, un DHT11 para la temperatura y humedad del ambiente, y un relé que activa una bomba de agua para regar.

El sistema riega automaticamente cuando la humedad del suelo baja del 50% y para cuando llega al 70%. Tambien se puede activar el riego de forma manual desde la API si alguien lo necesita. La bomba es chica, de 3 a 5 voltios, y el rele la protege.

5.4 Arduino Mega - Maestro

El Mega es el cerebro de todo. En cada ciclo del programa hace varias cosas en orden:

  • Revisa si el ESP32 le mando algun comando (como abrir puerta)
  • Lee los datos del esclavo 1 por I2C
  • Lee los datos del esclavo 2
  • Lee los datos del esclavo 3
  • Calcula si hay alguna condicion de alarma
  • Manda toda la informacion al ESP32 por el puerto serie
  • Si hay alarma, prende el buzzer unos 200 milisegundos
  • Actualiza la pantalla LCD con la informacion de cada zona, rotando cada 3 segundos

La pantalla LCD tiene direccion I2C 0x27 y es de 16 caracteres por 2 lineas. Muestra tres pantallas que rotan: una con los datos de servidores (humo, temperatura, humedad y si el ventilador esta prendido), otra con el estado de las puertas, y otra con los datos del jardin.

Para la alarma, el Mega revisa cuatro condiciones: si se detecto humo, si la temperatura de servidores paso los 30 grados, si la temperatura del jardin paso los 45 grados, o si hay un intruso (PIR detecta movimiento con las puertas cerradas). Si alguna se cumple, prende la alarma.

5.5 ESP32 - Gateway

El ESP32 recibe los datos del Mega por el puerto Serial2 a 9600 baudios. Los datos llegan en una sola linea con este formato:

Area 1: humo, temperatura, humedad, alerta, nivel ventilador | Area 2: btn1, btn2, pir, puerta1, puerta2, alerta puertas | Area 3: humedad suelo, temperatura, humedad aire

El ESP32 separa la linea por los pipes y extrae los valores de cada area. Solo manda los datos a la API si detecta que hubo un cambio desde la ultima medicion. Esto evita enviar informacion repetida.

Aparte, cada 2 segundos el ESP32 le pregunta a la API si hay comandos pendientes (como abrir una puerta). Si encuentra uno, lo manda al Mega con el formato "CMD,OPEN_DOOR_1" y marca el comando como ejecutado.

5.6 API y Base de Datos

La API esta hecha con Bun y el framework Elysia. Corre en el puerto 3000 de la maquina con IP 192.168.1.49. Recibe peticiones POST con los datos de cada area y las guarda en PostgreSQL. La base de datos esta en AWS RDS (Aurora PostgreSQL) con el host datacenterfk.cq3gc4ceq61b.us-east-1.rds.amazonaws.com.

Las tablas que usamos son: area_servidores (temperatura, humo, humedad, alerta, fan), area_puertas (btn1, btn2, pir, puerta1, puerta2, alerta), area_jardin (humedad_suelo, temperatura, humedad_aire), y otras para usuarios, comandos, accesos y visitas. La autenticacion usa JWT con roles de admin, gerente y empleado.

5.7 Telegram

Integramos un bot de Telegram para mandar notificaciones. La API tiene un endpoint POST /api/telegram/enviar que recibe un mensaje y lo manda al chat del administrador usando la API de Telegram. El token del bot y el ID del chat se configuran con variables de entorno.

Cuando el sistema detecta una alarma critica (humo o temperatura muy alta), puede llamar a este endpoint para avisar al instante. Para crear el bot solo se necesita hablar con @BotFather en Telegram, pedir un token nuevo, y ponerlo en el archivo .env junto con el chat ID.

6. Protocolo I2C

I2C significa Inter-Integrated Circuit y es un bus de comunicacion que invento Philips. Usa dos lineas: SDA para los datos y SCL para el reloj. Todos los dispositivos comparten las mismas lineas y cada uno tiene una direccion unica. El maestro es el que genera el reloj e inicia las comunicaciones.

La comunicacion funciona asi: el maestro manda una condicion de START, luego la direccion del esclavo con un bit que dice si va a leer o escribir, el esclavo responde con un ACK (reconocimiento), y ahi empieza la transferencia de datos. Al final, el maestro manda una condicion de STOP. Todo esto pasa a 100 kHz en nuestro sistema.

En Arduino usamos la libreria Wire.h. El maestro llama a Wire.requestFrom(direccion, cantidad) para pedir datos, y los esclavos usan Wire.onRequest(manejador) para registrar la funcion que envia los datos cuando el maestro pregunta. Para mandar comandos a los esclavos, el maestro usa Wire.beginTransmission(), Wire.write() y Wire.endTransmission().

Las direcciones I2C que usamos son:

DispositivoDireccionPara que
Esclavo 18 (0x08)Servidores: DHT11, MQ-135, ventilador
Esclavo 29 (0x09)Puertas: servos, botones, PIR, DHT11
Esclavo 310 (0x0A)Jardin: FC-28, DHT11, bomba
LCD 16x20x27Pantalla de estado en el Mega

Las conexiones fisicas son simples: el pin 20 del Mega va al A4 de cada Uno (SDA), el pin 21 va al A5 de cada Uno (SCL), y todos comparten GND. El LCD tambien se conecta a los mismos pines SDA y SCL. Aunque los modulos tienen resistencias pull-up, nosotros agregamos unas de 4.7 kiloohmios para mayor estabilidad.

El bus I2C tiene una limitacion: mientras mas dispositivos, mas se cargan las lineas y puede haber problemas de comunicacion. Con solo 4 dispositivos (3 esclavos + LCD) no tuvimos problemas, pero en teoria se pueden conectar hasta 127.

7. Reglas del Sistema y Prioridades

7.1 Como funciona el Mega

El Mega hace siempre el mismo ciclo en este orden:

  1. Fijarse si el ESP32 mando algun comando. Si es asi, lo procesa antes de seguir.
  2. Pedir datos al esclavo 1 por I2C, luego al 2, luego al 3. Si uno no responde, sigue con el siguiente.
  3. Decidir si hay alarma: revisa humo activo, temperatura de servidores mayor a 30, temperatura de jardin mayor a 45, o alerta de intruso.
  4. Mandar todos los datos al ESP32 por el puerto serie.
  5. Si hay alarma, hacer sonar el buzzer 200 milisegundos.
  6. Cambiar la pantalla del LCD si ya pasaron 3 segundos.

7.2 Prioridad de las Alarmas

Clasificamos las alarmas en cuatro niveles para saber que hacer en cada caso:

NivelCuando pasaQue hace el sistema
P1 - CriticaHumo detectado, o temperatura de servidores > 30, o jardin > 45Buzzer suena, LCD dice "ALARMA", manda Telegram
P2 - AltaPIR detecta movimiento con las dos puertas cerradasBuzzer suena, LCD dice "INTRUSO", manda Telegram
P3 - MediaTemperatura entre 25 y 29 gradosVentilador se pone en velocidad media (40%)
P4 - BajaTemperatura entre 20 y 24 gradosVentilador se pone en velocidad baja (20%)

7.3 Reglas Importantes

  • No saturar la base de datos: El ESP32 solo manda datos a la API cuando algo cambia. Si todo sigue igual, no manda nada.
  • Intruso condicional: La alarma de intruso solo se activa si el PIR detecta movimiento Y las puertas estan cerradas. Si una puerta esta abierta, el movimiento no genera alarma, solo se reporta como presencia.
  • Buzzer solo para P1 y P2: Las alarmas de temperatura mas bajas no hacen sonar el buzzer, solo mueven el ventilador.
  • Ventilador proporcional: La velocidad del ventilador depende de la temperatura, no es solo prender o apagar.
  • Los esclavos trabajan solos: Cada Uno lee sus sensores aunque el Mega no les pida datos. Si el Mega se desconecta, ellos siguen funcionando.
  • Linea base del MQ-135: El sensor de humo se calibra solo durante los primeros 30 segundos y se sigue ajustando con el tiempo para evitar falsas alarmas.
  • Puertas con temporizador: Las puertas se abren 5 segundos y se cierran solas, no quedan abiertas.
  • Pantalla rotativa: El LCD cambia de vista cada 3 segundos para mostrar servidores, puertas y jardin.

9. Resultados Obtenidos

9.1 Comunicacion I2C

Probamos la comunicacion entre el Mega y los tres Unos y funciona bien. El Mega puede leer los tres esclavos en menos de 200 milisegundos, lo que deja suficiente tiempo para el resto del ciclo.

9.2 Comunicacion UART

El TXS0108E funciona para pasar los datos de 5V a 3.3V y viceversa. A veces se pierde el primer byte cuando se conecta, pero el codigo ya tiene una forma de recuperarse: el Mega espera medio milisegundo antes de mandar datos, y el ESP32 puede detectar si la linea empieza con coma para arreglarla.

9.3 Deteccion de Humo

El algoritmo de linea base para el MQ-135 funciona. Se estabiliza en unos 30 segundos y el delta del 5% permite detectar humo sin que el cambio normal del aire dispare falsas alarmas.

9.4 Alarma de Intruso

La logica de PIR + puertas cerradas funciona bien. Si alguien se mueve con las puertas cerradas, suena la alarma y la pantalla muestra "INTRUSO". Si una puerta esta abierta, el movimiento se reporta normal sin alarma.

9.5 Ventilador PWM

El motor responde a los cuatro niveles de velocidad y no se calienta ni se quema con el PWM al 60%. La transicion entre velocidades es inmediata.

10. Conclusiones

El proyecto logro su objetivo principal: construir un sistema distribuido de monitoreo y control para un Data Center usando microcontroladores Arduino. La arquitectura maestro-esclavo con I2C funciona bien y demuestra en la practica como se comunican varios procesadores en un mismo bus.

El sistema de alarmas con prioridades (P1 a P4) es un ejemplo claro de como se manejan las interrupciones en un sistema embebido: el Mega evalua constantemente las condiciones y responde segun la gravedad del evento, igual que un controlador de interrupciones.

El control PWM del ventilador muestra como una senal digital puede controlar la potencia de un motor variando el ciclo de trabajo, que es uno de los temas clave del curso.

La conexion con AWS (aunque sea via API REST y no directo por MQTT) demuestra la jerarquia de memoria del sistema: los datos viajan desde la RAM del microcontrolador hasta la nube, pasando por I2C, UART y WiFi.

El protocolo I2C funciono sin problemas con los 4 dispositivos en el bus (3 esclavos + LCD). Es un protocolo simple pero efectivo para sistemas embebidos donde no se necesitan velocidades muy altas.

10.1 Cosas que faltaron o se podrian mejorar

  • No pudimos implementar el sensor INA219 para medir consumo de energia, asi que el modo de ahorro por bateria baja no esta disponible.
  • Tampoco tenemos el modulo Bluetooth HC-04 que estaba en el diseno original.
  • Los paneles solares que tenemos generan poca energia, no son suficientes para alimentar todo el sistema.
  • El acceso por NFC funciona con stickers y verificacion por API, no con modulo RFID fisico como se habia planeado.
  • Falta implementar MQTT para conectar directamente con AWS IoT Core, por ahora usamos HTTPS.
  • Se podria agregar un watchdog por hardware para que el Mega se reinicie solo si se cuelga.

D. El bus I2C en la practica

Las conexiones del I2C son asi: el SDA del Mega (pin 20) va al A4 de cada Uno y al SDA del LCD. El SCL del Mega (pin 21) va al A5 de cada Uno y al SCL del LCD. Todo comparte GND. Ponemos resistencias de 4.7 kiloohmios entre SDA y 5V, y entre SCL y 5V, para que la senal se mantenga estable. El bus trabaja a 5V porque el Mega y los Unos son de 5V, y el LCD tambien soporta 5V.
