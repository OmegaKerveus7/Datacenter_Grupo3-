1. Introducción

El presente documento constituye el manual técnico del sistema DataCenter Access Control, desarrollado como proyecto final del curso de Arquitectura de Computadoras por el Grupo 3. El sistema integra hardware embebido (Arduinos, sensores, actuadores) con una plataforma web moderna para el control de acceso y monitoreo ambiental de un centro de datos a escala educativa.

El proyecto aborda conceptos fundamentales de la arquitectura de computadoras: comunicación entre periféricos vía I2C y UART, jerarquía maestro-esclavo, manejo de interrupciones, conversión analógico-digital, control PWM, y la integración de sistemas embebidos con servicios en la nube. Se implementa un sistema de control de acceso de dos puertas: la Puerta 1 con autenticación remota simple y la Puerta 2 mediante verificación NFC asociada a visitas programadas.

El sistema se compone de tres microcontroladores esclavos (Arduino Uno) conectados vía I2C a un maestro (Arduino Mega), el cual se comunica mediante UART a un ESP32 que funge como gateway hacia una API REST alojada en AWS. La interfaz de usuario es una aplicación web desarrollada con React y Vite, servida mediante Bun.

2. Objetivos

2.1. Objetivo General

Diseñar e implementar un sistema integral de control de acceso y monitoreo ambiental para un centro de datos, aplicando los conceptos de arquitectura de computadoras, comunicación entre periféricos y sistemas embebidos.

2.2. Objetivos Específicos

  1. Implementar una red I2C con un maestro (Arduino Mega) y tres esclavos (Arduino Uno) para la adquisición distribuida de sensores.
  2. Desarrollar un sistema de control de acceso con dos niveles: remoto (Puerta 1) y NFC con visitas programadas (Puerta 2).
  3. Implementar un sistema de alertas escalonado por prioridades con notificaciones vía Telegram y correo electrónico.
  4. Diseñar una interfaz web con roles de usuario (empleado, gerente, admin) para administración y monitoreo.
  5. Documentar el mapa completo de interrupciones del microcontrolador y su aplicación en el sistema.

3.2. Flujo de datos

El flujo de información sigue una arquitectura jerárquica de tres niveles:

  1. Nivel sensor (adquisición): Cada esclavo Arduino lee sus sensores periódicamente y almacena los datos en un buffer. Cuando el maestro lo solicita por I2C, transmite la cadena formateada.
  2. Nivel de procesamiento local: El Mega recopila los datos de los tres esclavos, calcula la alerta global, controla el buzzer y la LCD, y envía todo al ESP32 por UART.
  3. Nivel de nube: El ESP32 parsea los datos, detecta cambios y envía HTTP POST al backend, que persiste en PostgreSQL y dispara notificaciones si hay alertas.

El polling de comandos sigue el camino inverso: el ESP32 consulta cada 2 segundos GET /api/comandos/pendientes, y si hay comandos (ej: OPEN_DOOR_1), los envía al Mega por UART, que los reenvía al esclavo 2 por I2C.

Control de acceso por botones

Los botones físicos del esclavo 2 permiten abrir las puertas localmente. Sin embargo, cuando el sistema detecta una intrusión (PIR activo con ambas puertas cerradas), el Mega envía el comando 'L' (lock) que bloquea los botones, impidiendo que un intruso pueda abrir las puertas y escapar. Los comandos remotos 'O' (open) y 'C' (close) provenientes del Mega siempre funcionan, incluso en estado bloqueado, para permitir que el personal autorizado pueda controlar las puertas.

Detección PIR

El sensor PIR tiene un debounce de 200 ms, mantiene la señal activa por 1 segundo, y tiene un tiempo de rearme de 6 segundos. Esto evita falsos positivos por ruido eléctrico o cambios bruscos de temperatura.

Puertas servo-controladas

Cada puerta utiliza un servo motor con dos posiciones: cerrada (0°) y abierta (95° para puerta 1, 90° para puerta 2). Tras 5 segundos en posición abierta, la puerta se cierra automáticamente.

4.4. Arduino Mega — Maestro I2C

El Arduino Mega 2560 actúa como maestro del bus I2C y concentrador de datos. Sus funciones principales son:

  • Lectura cíclica: Cada ~2 segundos solicita datos a los tres esclavos (direcciones 8, 9, 10).
  • Cálculo de alerta global: Combina las condiciones de todos los esclavos para determinar si hay una situación crítica.
  • Control de buzzer: Activa el buzzer en el pin 8 durante 5 segundos ante cualquier alerta.
  • Display LCD: Muestra alternadamente los datos de Servidores y Jardín en un LCD 16x2 con dirección I2C 0x27.
  • Reenvío de comandos: Recibe comandos del ESP32 por Serial1 y los reenvía al esclavo 2 por I2C.
  • Bloqueo de seguridad: En caso de intrusión, envía comando 'L' al esclavo 2 para bloquear botones.
  • Cierre de emergencia: En caso de humo/incendio, cierra ambas puertas automáticamente (sin bloquear).

Cálculo de alerta global

alerta = (humo == 1 || tempServidores > 30 || tempJardin > 45 || alertaPuertas == 1) ? 1 : 0;

alertaPuertas = (pir == 1 && puerta1 == 0 && puerta2 == 0) ? 1 : 0;

La alerta se mantiene activa durante 5 segundos después de que la condición desaparezca, para evitar parpadeo del buzzer.

4.5. ESP32 — Gateway WiFi

El ESP32 actúa como puente entre el Arduino Mega (UART) y el backend (HTTP). Funciones clave:

  • Recepción UART: Lee datos del Mega a 9600 baudios por Serial2 (pines RX2=16, TX2=17).
  • Detección de cambios: Solo envía datos al backend si algún valor cambió, evitando escrituras innecesarias en la base de datos.
  • HTTP POST: Envía datos a tres endpoints: /area/servidores, /area/puertas, /area/jardin.
  • Polling de comandos: Cada 2 segundos consulta GET /api/comandos/pendientes y ejecuta los comandos pendientes.
  • Watchdog: Configuración en caliente — si el ESP32 se congela, desconectar y reconectar el USB lo restablece sin pérdida de datos.

5. Protocolos de Comunicación

5.1. I2C entre Arduinos

El bus I2C opera a 100 kHz (modo estándar). El Arduino Mega es el único maestro; los tres esclavos tienen direcciones fijas:

DispositivoDirección I2CRol
Arduino MegaMaestroInicia todas las comunicaciones
Esclavo 10x08Sala de servidores
Esclavo 20x09Puertas y acceso
Esclavo 30x0AJardín y riego

El maestro utiliza Wire.requestFrom(dir, 20) para solicitar hasta 20 bytes de cada esclavo. Los esclavos responden con la función Wire.onRequest(), enviando una cadena delimitada por pipes y comas. El maestro también puede escribir datos a los esclavos mediante Wire.beginTransmission() + Wire.write() + Wire.endTransmission(), usado para enviar comandos de apertura/cierre de puertas y bloqueo de seguridad.

5.2. UART entre Mega y ESP32

La comunicación entre el Mega y el ESP32 se realiza mediante UART a 9600 baudios, 8 bits de datos, sin paridad, 1 bit de parada. El Mega transmite por Serial1 (pines 18 TX1 / 19 RX1) y el ESP32 recibe por Serial2 (pines 16 RX2 / 17 TX2). Se utiliza un módulo TXS0108E como adaptador de nivel lógico (5V del Mega a 3.3V del ESP32).

El formato de trama es:

1,{humo},{temp},{humedad},{alerta},{fan}|2,{btn1},{btn2},{pir},{p1},{p2},{alertaPuertas}|3,{suelo},{tempJ},{humJ}\n

El ESP32 también envía comandos al Mega con el formato CMD,{comando}\n. El Mega parsea estos comandos y los reenvía por I2C al esclavo correspondiente.

5.3. HTTP/REST entre ESP32 y Backend

El ESP32 se conecta a la red WiFi "Omega" y realiza peticiones HTTP al backend en http://192.168.1.49:3000. Se utiliza el formato JSON con Content-Type application/json. La detección de cambios evita enviar datos duplicados, reduciendo el tráfico y las escrituras en base de datos.

5.4.2. Implementación actual (polling)

En la versión actual, el sistema no utiliza interrupciones hardware. Todo el control se realiza mediante polling con la función millis() para temporización y delay() para pausas. El ciclo principal del Mega tiene una duración aproximada de 2 segundos, determinada por el delay(2000) al final del loop. Esto es suficiente para la tasa de actualización requerida por los sensores ambientales, pero introduce una latencia de hasta 2 segundos en la detección de eventos críticos como una intrusión.

5.4.3. Rediseño propuesto con interrupciones

Para cumplir con los requisitos del curso y mejorar la capacidad de respuesta del sistema, se propone el siguiente rediseño basado en interrupciones:

INT0 — PIR (flanco ascendente)

ISR(INT0_vect) {
    pirFlag = true;  // Solo actualiza bandera
}

// En loop():
if (pirFlag) {
    pirFlag = false;
    alertaPuertas = (puerta1 == 0 && puerta2 == 0) ? 1 : 0;
}

Latencia estimada: < 10 µs desde el flanco del PIR hasta la activación de la bandera.

Timer1 — Muestreo periódico

ISR(TIMER1_COMPA_vect) {
    timerFlag = true;  // Bandera para el loop
}

// Configuración: CTC mode, 500 Hz → periodo 2 ms

Esto permite ejecutar las lecturas de sensores cada 500 ms de forma independiente a la duración del loop principal.

UART1 RX — Recepción de ESP32

ISR(USART1_RX_vect) {
    char c = UDR1;              // Leer byte recibido
    if (c == '\n') {
        cmdReadyFlag = true;    // Trama completa lista
    } else if (idx < BUFFER_SIZE - 1) {
        cmdBuffer[idx++] = c;   // Almacenar en buffer circular
    }
}
// En loop():
if (cmdReadyFlag) {
    cmdReadyFlag = false;
    procesarComando(cmdBuffer);
    idx = 0;
}

5.4.4. Prioridades de interrupción

El ATmega2560 del Arduino Mega define una tabla de vectores de interrupción con prioridades fijas. Las interrupciones externas (INT0-INT7) tienen prioridad sobre las interrupciones de timer. La prioridad relativa es:

  1. Reset (prioridad más alta)
  2. INT0 – INT7 (interrupciones externas)
  3. Timer1 COMPA (timer compare match)
  4. USART1 RX (recepción UART)
  5. SPI, TWI (I2C), etc.

Se recomienda asignar INT0 al PIR por ser el evento más crítico en términos de latencia (intrusión debe detectarse en < 10 µs). Timer1 se asigna al muestreo periódico (500 Hz), y UART1 RX a la recepción de datos del ESP32. Las ISR deben ser mínimas: solo actualizan banderas. La lógica de negocio se ejecuta en el loop principal.

8. Seguridad y Control de Acceso

8.1. Autenticación

El sistema utiliza JSON Web Tokens (JWT) para la autenticación de usuarios. El token se genera en el login y debe incluirse en el header Authorization: Bearer {token} de todas las peticiones autenticadas. La clave secreta es datacenter-fk-secret-key-2026.

8.2. Control de Roles

Cada usuario tiene un rol (empleado, gerente, admin) que determina las operaciones permitidas. El middleware rolGuard() en el backend verifica que el usuario tenga el rol requerido para acceder a cada endpoint. En el frontend, el componente ProtectedRoute filtra las rutas por rol.

8.3. Bloqueo de seguridad por intrusión

Cuando el PIR detecta movimiento con ambas puertas cerradas (posible intruso dentro del DataCenter), el Mega envía el comando 'L' al esclavo 2, que bloquea los botones físicos de apertura. Esto impide que un intruso pueda salir del recinto. Los comandos remotos desde el Dashboard siguen funcionando para que el personal autorizado pueda liberar la situación. En caso de humo o incendio, las puertas NO se bloquean para permitir la evacuación.

8.4. Acceso por NFC

La Puerta 2 utiliza un chip NFC con URL fija. El chip almacena la URL http://192.168.1.49:5173/verificar-nfc?key=7C:B3:70:51. El key está hardcodeado en el servidor (PUERTA2_KEY) y solo identifica la puerta, no al usuario. La identificación del usuario se realiza mediante el formulario de login en la página de verificación NFC. Las visitas tienen una ventana de 10 minutos desde la hora programada; pasado ese tiempo, la visita vence automáticamente.

10. Conclusiones

  1. La implementación de una red I2C con un maestro y tres esclavos demostró ser una solución efectiva y de bajo costo para la adquisición distribuida de sensores en un centro de datos a escala educativa, con una tasa de actualización de 2 segundos suficiente para monitoreo ambiental.
  2. El sistema de control de acceso con dos niveles (remoto y NFC con visitas programadas) cumple con los requisitos de seguridad del DataCenter, permitiendo diferenciar entre acceso administrativo (Puerta 1) y acceso de visitas (Puerta 2) con verificación en ventana de tiempo.
  3. La integración de sistemas embebidos (Arduino) con servicios en la nube (AWS RDS, Telegram Bot, Email SMTP) mediante un gateway ESP32 proporciona una arquitectura escalable que puede extenderse a aplicaciones industriales de IoT.
  4. La implementación de roles de usuario (empleado, gerente, admin) con control de acceso basado en JWT y middleware de verificación garantiza que cada operación del sistema sea realizada por personal autorizado según su nivel de responsabilidad.
  5. El análisis de latencia del sistema de alertas muestra que con el rediseño basado en interrupciones propuesto se puede reducir el tiempo de respuesta a eventos críticos de 2 segundos (polling) a menos de 500 ms, cumpliendo con los requisitos del curso de Arquitectura de Computadoras.
  6. El mapeo completo de interrupciones del ATmega2560 demuestra la aplicación práctica de los conceptos teóricos de vectores de interrupción, ISR mínimas, prioridades y latencia en un sistema embebido real.
